## Problem Statement

Catalog quality work is repetitive and rules-based, but publishing mistakes can
directly affect conversion and compliance.

## Trigger

Run when a new supplier feed or internal product batch enters QA.

## Inputs

- product attributes
- taxonomy rules
- merchandising standards

## Systems/Tools

- PIM
- feed source
- QA checklist

## Workflow Steps

1. Check products for missing, malformed, or conflicting fields.
2. Suggest normalized attributes and enrichment candidates.
3. Draft a publish-readiness summary with flagged issues.
4. Send the package for merchandiser review.

## Human Approval Points

- Merchandiser approves any publish-facing changes.
- Regulated claims are checked against source facts.

## Outputs

- QA issue list
- enrichment suggestions
- readiness summary

## Success Metrics

- faster QA cycles
- fewer missing attributes
- better acceptance of enrichment drafts

## Failure Modes

- wrong attribute inference
- inconsistent taxonomy mapping
- generated language exceeds policy

## Security Notes

- keep source data in approved product systems
- retain traceability for all sensitive product claims

## Token-Cost Notes

- prefer structured specs to raw feed text
- split validation and copy tasks to reduce waste

## Deployment Mode

`Hybrid` works because the machine can handle rule-heavy QA, while publishing
decisions and final copy remain human-owned.

## Prompt Skeleton

```text
Role: Catalog QA assistant

Objective: Identify catalog quality issues and suggest reviewable enrichment.

Inputs: Product specs, taxonomy rules, merchandising checklist.

Constraints: Do not invent product facts. Flag any regulated or uncertain claim for review.

Required checks: Separate missing data from inferred enrichment.

Output format: Quality issues, suggested attributes, readiness summary, review flags.
```
