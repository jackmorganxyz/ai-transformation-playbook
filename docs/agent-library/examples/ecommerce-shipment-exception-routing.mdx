## Problem Statement

Shipment and returns exceptions are expensive because people spend time deciding
where the case belongs before anyone fixes it.

## Trigger

Run whenever a qualifying exception event arrives in the operations queue.

## Inputs

- exception event data
- order context
- routing and handling rules

## Systems/Tools

- operations dashboard
- OMS or commerce platform
- carrier or warehouse feed

## Workflow Steps

1. Interpret the latest exception event with current order context.
2. Classify the likely issue type and urgency.
3. Draft a routed summary and recommend the owning queue.
4. Flag cases with low confidence or irreversible consequences.

## Human Approval Points

- Specialist confirms the route for ambiguous cases.
- Any financial or inventory action is reviewed before execution.

## Outputs

- exception classification
- routed summary
- recommended action

## Success Metrics

- faster routing
- higher queue accuracy
- lower exception aging

## Failure Modes

- event noise causes wrong routing
- low-confidence cases are treated as certain
- irreversible actions are suggested too early

## Security Notes

- confine order and operations data to approved systems
- log approvals for any downstream action

## Token-Cost Notes

- use structured events first
- only expand context for unresolved or ambiguous exceptions

## Deployment Mode

`Hybrid` keeps classification efficient while reserving meaningful actions for
human approval.

## Prompt Skeleton

```text
Role: Operations routing assistant

Objective: Classify a shipment exception and prepare a routed summary.

Inputs: Event payload, order context, routing rules.

Constraints: Flag low confidence, avoid executing actions, and separate operational facts from suggestions.

Required checks: Highlight any case that could affect refunds, inventory, or customer promises.

Output format: Exception type, urgency, recommended queue, case summary, review flags.
```
