## Problem Statement

Returns-related tickets need policy grounding, order facts, and routing logic.
Most of the time cost is in gathering that context.

## Trigger

Run when a return, exchange, or refund issue is opened.

## Inputs

- order and shipment details
- return status
- customer message

## Systems/Tools

- help desk
- commerce platform
- returns system

## Workflow Steps

1. Pull order and return status for the relevant issue.
2. Match the case against current policy rules.
3. Draft a case summary and recommended queue or action.
4. Flag exceptions for human review.

## Human Approval Points

- Agent approves every customer-facing outcome.
- Refund or fraud exceptions are escalated.

## Outputs

- issue classification
- case summary
- recommended routing

## Success Metrics

- faster triage
- correct routing
- reduced lookup time

## Failure Modes

- wrong policy applied
- incomplete order context
- missed fraud signal

## Security Notes

- restrict access to order and payment systems
- keep refund actions human-approved

## Token-Cost Notes

- use structured fields first
- retrieve only the policy snippet needed for the current issue

## Deployment Mode

`Hybrid` is a fit because policy-backed prep is automatable, but financial and
customer-impacting actions still need review.

## Prompt Skeleton

```text
Role: Returns operations assistant

Objective: Classify a returns issue and prepare a policy-backed summary for review.

Inputs: Order data, return status, customer issue, applicable policy excerpt.

Constraints: Flag fraud or policy exceptions, avoid refund execution, and keep uncertainty visible.

Required checks: Highlight anything that requires supervisor or specialist review.

Output format: Issue type, case summary, recommended route, required approvals.
```
