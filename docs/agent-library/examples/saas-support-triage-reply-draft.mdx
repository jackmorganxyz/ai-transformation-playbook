## Problem Statement

Support teams need faster triage and draft generation without turning sensitive
customer issues into auto-pilot workflows.

## Trigger

Run on ticket creation, major ticket updates, or escalation requests.

## Inputs

- ticket text and metadata
- related account context
- approved support content

## Systems/Tools

- help desk
- knowledge base
- CRM

## Workflow Steps

1. Classify the ticket by intent, urgency, and policy sensitivity.
2. Retrieve the smallest set of relevant support content.
3. Draft a reply or an escalation summary.
4. Flag routing and confidence for the human reviewer.

## Human Approval Points

- Support agent approves the reply draft.
- Sensitive classes are routed to the right specialist queue.

## Outputs

- triage label
- reply draft
- escalation summary

## Success Metrics

- faster first response
- accurate routing
- reduced manual summarization time

## Failure Modes

- unsafe reply for a sensitive issue
- missed urgency on outage or access cases
- overlong context inflates cost with little quality gain

## Security Notes

- keep all customer information inside approved systems
- preserve escalation traces for audited issue classes

## Token-Cost Notes

- restrict retrieval to the active issue and top matching articles
- avoid sending full ticket history when a concise summary exists

## Deployment Mode

`Hybrid` is the default because quality is useful at draft time, but the
customer-facing step stays under agent review.

## Prompt Skeleton

```text
Role: Support triage assistant

Objective: Classify the ticket and draft a grounded reply or escalation summary.

Inputs: Ticket thread, metadata, account context, approved support content.

Constraints: Escalate sensitive classes, cite support content, and avoid promises not present in policy.

Required checks: Mark any case that needs specialist review before response.

Output format: Intent, urgency, recommended queue, draft reply, escalation summary.
```
